drm component framework based on rk3399

参考链接：https://blog.csdn.net/shikivs/article/details/103591971

1 component 框架简析

drm 使用component框架加载各个组件，component 框架中的设备包含两种类型—master device和component device，后者由前者管理，当所有的component device按照顺序初始化完成之后，master device才能进行最终的初始化。

master device在设备树中往往体现为一个子系统节点，比如XXX_subsystem，此节点下包含ports属性，ports属性中包含与master device关联的component device，以rk3399 的display_subsystem为例：

  • master device在设备树中的节点为：

    上述节点的ports 属性对应另外两个设备节点，分别为vopl_out和vopb_out，vopl和vopb是rk平台中的两个vop(video out processor)，前者最高支持2K输出(lite)，后者最高支持4K输出(big)。

  • component device在设备树中的节点为（以vopl为例）：

    在drm框架中，将vopl_out抽象为crtc设备，其子节点中包含许多remote节点,包括dsi、hdmi、edp等，这些设备在drm框架中被抽象为encoder设备，vopl通过这些remote节点与它们进行关联。

2 master device驱动代码解析

master device驱动源码文件路径为drivers/gpu/drm/rockchip/rockchip_drm_drv.c

2.1 驱动流程

  • 驱动流程在rockchip_drm_platform_probe(struct platform_device *pdv)中展开
  • 核心数据结构为struct component_match *match;,其实质是一个数组指针

  1. 获取master节点下的ports属性(即vopl_out和vopb_out节点)，然后获取这两个节点的父节点vopb和vopl节点，即为DRM框架中的CRTC设备

  2. 对于上面每一个有效的crtc设备，根据其设备节点中的endpoint节点，获取encoder设备

  3. 添加背光节点-backlight component

    实际上rk的dispaly_subsystem中并未定义backlight节点，但此段代码也未作错误退出处理

  4. match配置完成之后，注册一个带match的master device

    @dev: matser 设备
    @rockchip_drm_ops: master device的回调函数，主要为add_components、bind和unbind函数，当master下所有的component device初始化完成后，master device调用bind函数完成自身的初始化。

2.2 数据结构

(1) struct component_match

alloc和num用于内存的分配和管理
compare用于master与component的匹配，使用一个零长数组进行定义，可动态分配数组内存。

(2) struct master

component framwork中使用一个master_list来维护所有的master _device,在注册master设备时，将master device注册至此链表中

(3) struct component_master_ops rockchip_drm_ops

master 的回调函数 由于rk注册带match的master device，因此未实现add_components函数

2.3 核心函数

(1) component_match_add

(2) compare_of

自定义的匹配函数，在调用component_match_add时传入，作为match中compare的fn成员，要求设备节点与match中的compare[n]->data相同。

(3) rockchip_add_endpoints

(4) component_master_add_with_match

此函数的主要作用是将master device注册至component 框架中

(5) try_to_bring_up_master

3 DRM device 初始化

在master设备调用.bind函数时，将会完成drm的初始化

3.1 初始化流程

drm围绕着3个核心数据结构展开

  1. 实例化一个drm_device

  2. 实例化一个private对象，并根据master设备树节点中的属性对其某些成员进行初始化

  3. drm初始化iommu

    -| rockchip_drm_init_iommu(drm_dev);
    —| private->domain = iommu_domain_alloc(&platform_bus_type);
    —| drm_mm_init(&private->mm, start, end - start + 1);
    —| iommu_set_fault_handler(private->domain, rockchip_drm_fault_handler,
    drm_dev);

  4. drm初始化modeconfig

  5. 创建peoperties，初始化private中的各个property

    -| rockchip_drm_create_properties(drm_dev)
    —| drm_property_create_xxx
    —| drm_mode_create_tv_properties(drm_dev, 0, NULL)

  6. 尝试绑定所有的components -关联3.3(2)

  7. 设置connector的property -关联3.3(3)

  8. 初始化VBLANK

    Atomic 操作依赖 VSYNC 中断（即 VBLANK 事件），因此需要使用 hrtimer 来提供软件中断信号。在驱动初始化时调用 **drm_vblank_init()**，在 VSYNC 中断处理函数中调用 **drm_handle_vblank()**。
    crtc 即为display pipeline，需初始化vblank使其支持num_crtc display_crtc

  9. reset mode config

    调用各个component的reset回调函数

    -| drm_mode_config_reset(drm_dev);
    —|plane->funcs->reset(plane);
    —|crtc->funcs->reset(crtc);
    —|encoder->funcs->reset(encoder);
    —|connector->funcs->reset(connector);

  10. 初始化gem、fbdev和crtc

  11. 注册drm设备

3.2 数据结构

(1) struct drm_device *drm_dev

(2) struct drm_driver rockchip_drm_driver

每个DRM驱动程序的核心层是一个struct drm_driver 结构。驱动程序通常静态地初始化一个drm_driver，然后将其传递给drm_dev_alloc()来分配一个设备实例。设备实例完全初始化之后，可以使用drm_dev_register()对其进行注册，而后可以从用户空间访问此drm设备。
struct drm_driver结构包含描述驱动程序及其支持特性的静态信息，以及一些需要实现的函数指针供将来调用。
